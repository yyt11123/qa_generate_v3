# 香港保险顾问 AI 助手对话分析报告

> 基于 2052 条真实顾问会话的提问类型、提问特点与回答质量分析

---

## 1. 分析目的与数据范围

本报告基于某保险代理机构（org_code: AWM）AI 助手的真实使用日志，分析保险顾问在业务中的提问类型与提问特点，对回答失败的会话做根因诊断，并据此提出有针对性的优化方向。

**数据规模**

| 维度 | 数量 |
|---|---:|
| 原始会话（导出时间 2026-05-25） | 2052 |
| 剔除空会话（message_count = 0）后**有效会话** | **1954** |
| 总消息数（有效会话；空会话本身无消息） | 5009（user 2514 / assistant 2495） |
| 单轮（一问一答）会话 | 1795（91.9%） |
| 多轮（≥2 轮）会话 | 159（8.1%） |
| 检索失败会话 | 305（15.6%） |
| 涉及保险公司 | 19 家（命名归一化后） |
| 可识别身份的活跃顾问 | 65 人（约八成会话未带身份标签） |

**数据完整性**：每个会话带 `question_types`（粗分类）、`sub_types`（子类）、`company_names`（公司）、`product_names`（产品）等多值标签字段，以及失败标记 `has_failed_retrieval`。详情 JSON 含 7 步流水线节点结果（输入分类 → query 改写 → 召回 → 相关性过滤 → 生成），可用于失败定位。

**渠道分布**：企业微信 85.0%、小程序 9.3%、Web 5.6%。AI 助手主要嵌在企业微信顾问工作台中。

**隐私处理**：会话原始数据中含顾问姓名、电话、银行账户、牌照号、微信 OpenID 等个人字段，分析过程中均未读取或写入聚合输出文件，所有引用的顾问原句已经过姓氏 + 称谓、长数字串、邮箱的脱敏处理。

---

## 2. 分析方法

1. **第一阶段——确定性分布统计**：基于 sessions_list.json 的标签字段做频次与交叉。多值字段（question_types、company_names 等）按 explode 计数；公司命名做归一化合并（如「万通人寿」合并入「万通保险」）。
2. **第二阶段——语义分析**：读取有效会话的首问文本，并对失败会话、多轮会话进一步读取完整流水线节点结果。基于规则对首问做多标签意图分类（13 类），与系统已有的 5 类粗分类形成两层视角。
3. **失败诊断**：对 305 个失败会话遍历 `nodes_history`，定位卡在哪一步——召回为空 / 召回了但全过滤掉 / 有文档仍判失败 / 未走检索分支。两套节点前缀（`simpleQA_*` 与 `discountQA_*`）兼容处理。
4. **样本量警示**：分公司、分产品下钻时，n < 50 的视为小样本，结论标注「值得关注但非稳定」。

## 3. 顾问问什么——问题类型分布

### 3.1 系统粗分类（5 大类覆盖 99% 流量）

![问题类型分布](charts/question_types.png)

| 问题类型 | 数量 | 占比 |
|---|---:|---:|
| 单业务细节咨询 | 1170 | 59.9% |
| 优惠信息咨询 | 391 | 20.0% |
| 多业务对比 | 222 | 11.4% |
| 闲聊 | 205 | 10.5% |
| 反向询问类 | 52 | 2.7% |
| 缴费指引/售后/其他 5 类 | 12 | < 1% |

**核心结论**：六成提问聚焦"某个具体产品"的细节，第二大类是「优惠信息」。这两类合计 80%，是 AI 助手的主战场；「多业务对比」虽然只占 11%，却是后续失败分析中需要重点关注的难题。

### 3.2 保险公司分布

![公司分布](charts/companies.png)

头部 5 家——友邦 19.1%、保诚 13.3%、万通 11.9%、宏利 10.6%、永明 7.8%——合计占有标签会话的 62.7%（分母为带公司标签的会话）。其余 14 家中小公司分布较散，多家样本量在 30~90 区间。

### 3.3 意图细类（在粗分类之上的更细划分）

基于 1954 条首问原文，规则法多标签分类（一条会话可命中多类，因此各项占比之和大于 100%；占比口径为"命中该意图的会话数 ÷ 1954"）：

| 意图细类 | 占比 |
|---|---:|
| 查产品/概念详情（"X 是什么"、"介绍下 Y"） | 35.0% |
| 问优惠/活动（折扣、预缴、回赠） | 22.4% |
| 多产品/多公司对比 | 12.1% |
| 要具体数字/案例演示（保费、收益、IRR） | 10.3% |
| 招呼/闲聊 | 10.2% |
| 查核保结论/健康告知 | 8.6% |
| 要产品资料/文件（PDF、计划书、培训手册） | 7.6% |
| 确认资格/投保门槛（年龄、居住地、非内地客） | 7.0% |
| 投保/缴费操作指引 | 4.5% |
| 反向询问 | 2.4% |
| 要投保建议/推荐方案 | 2.1% |
| 售后/理赔 | 2.1% |
| 其他/未匹配 | 17.3% |

> 未匹配率 17.3%：未被规则覆盖的多为口语化、跨主题或场景较罕见的表达，后续可用语义聚类做交叉验证。

**在粗分类基础上多看到的真实需求**：
- **「核保结论/健康告知」（8.6%）**：顾问在面对客户病史时需要快速判断"能不能投保、要不要加费"。这类问题专业性强，常见话题涉及妇科疾病、结节类、肝病、儿童疾病等类别。
- **「要产品资料/文件」（7.6%）**：顾问把 AI 助手当作产品资料库使用，索取宣讲版 PDF、计划书、培训手册等。这是确定性的"取资料"路径，与"求解释"是两类不同的回答需求。
- **「投保建议/推荐」（2.1%）**：把 AI 当方案规划助手，让其根据客户画像（如"42 岁女性，预算 5w 美金，要终身寿险"）反向推荐产品。

### 3.4 一句话总结

顾问的提问以"具体某款产品的细节"为主轴，次之是"优惠"，再次之是核保、对比、要资料、要数字。AI 助手承担了产品百科 + 优惠速查 + 核保决策辅助 + 资料分发 这 4 个角色。

## 4. 顾问怎么问——提问特点

基于全部 1954 条有效会话的首条 user 消息原文。

### 4.1 提问偏短，长问罕见

| 长度档 | 数量 | 占比 |
|---|---:|---:|
| 短问（< 15 字） | 724 | 37.1% |
| 中等（15~50 字） | 1136 | 58.1% |
| 长问（> 50 字） | 94 | 4.8% |

**95% 以上是中短句**，长问只占 5%。这意味着顾问倾向用关键词快速触发问题——「宏挚传承预缴优惠」「万通的续保宽限期」——而非先把客户背景完整描述清楚再提问。这种"短而抽象"的提问方式直接影响了 AI 能掌握多少前置上下文。

### 4.2 八成以上首问不带客户场景信息（关键发现）

仅 **15.7%** 的首问含具体客户参数（年龄、保额、缴费年期、币种、病史 任一）。明细：

| 场景参数 | 含此参数的会话占比 |
|---|---:|
| 缴费年期（如「5 年缴」） | 6.0% |
| 币种（港币/美金/人民币） | 5.2% |
| 年龄（如「42 岁女」） | 3.8% |
| 病史/健康（具体疾病名） | 3.3% |
| 保额（如「100 万美金」） | 2.5% |

**这是影响 AI 回答精准度的最主要使用习惯**。当顾问只问"宏挚传承的产品介绍"而不告诉 AI 客户是 35 岁还是 60 岁、预算多少、币种偏好，AI 只能返回通用产品资料，难以做到"刚好匹配这个客户的方案"。

少数带完整场景的长问质量很高，例如：

> 「客人想买一份香港消费型高端住院险保单，两名未成年子女分别在英国和亚太地区，均为非吸烟者，客人比较青睐保诚，请帮忙推荐适合两人的保单。同步推荐一份其它保险公司的以做比较。」

这种问题包含家庭结构、地区、风险因子、偏好公司、对比需求，AI 才有能力给出实用回答。但这类提问只占极少数。

### 4.3 句式：以疑问句为主，多问题塞一句很少

| 句式特征 | 占比 |
|---|---:|
| 疑问句（含疑问词或问号） | 57.9% |
| 祈使句（"发我..."、"查一下..."、动词开头无问号） | 5.4% |
| 一句塞 ≥ 2 个问题 | 2.4% |
| 明确索取产品资料/文件 | 7.6% |

少有"一次问 N 个问题"的复杂提问；但有 **7.6% 是确定性的「取资料」诉求**——这一类与"求解释"的回答路径完全不同，理论上可以通过资料推送而非长答案直接满足。

### 4.4 语言风格：99% 普通话/简体书面语

| 风格特征 | 占比 |
|---|---:|
| 含繁体字符 | 2.7% |
| 含粤语口语字（嘅/咗/咁/喺/啲 等） | 0.7% |
| 含礼貌用语（请/麻烦/谢谢） | 8.0% |

虽然业务在香港，但 99.3% 的提问是普通话或简体书面语，几乎不出现粤语口语。AI 助手按普通话理解和应答即可，无需为粤语口语专门优化。

### 4.5 提问特点总结

> 「**短问、抽象、关键词式、带强烈职业语境（公司+产品名）**」是顾问提问的主要画像。
>
> 顾问知道客户故事，但很少把客户故事完整搬给 AI；他们更习惯用 30 秒拼一个关键词触发问题，AI 拿到的是"信息浓缩到极限"的输入。这一使用习惯是后续 AI 助手优化的最重要前提。

## 5. 检索失败根因诊断

305 条失败会话占有效会话 15.6%。把每个失败会话的流水线节点结果展开做归因，得到三个业务结论：

![按问题类型的失败率](charts/failure_by_question_type.png)

**结论一：失败的主要矛盾在生成环节而非召回。** 在 305 个失败会话里，**55% 属于"已经检索到相关文档、相关性过滤也通过了，但最终仍判失败"——也就是问题不在"找不到资料"，而在"找到了没用上"**。相对地，"知识库根本没有可用文档"只占 8%。这与"答不上来等于知识库没盖到"的直觉相反，是本次分析最值得强调的发现。

**结论二：失败率最高的多产品对比类（26.6%）和反向询问类（32.7%），失败也几乎全部集中在生成环节**。这两类问题对应的检索都正常，是生成阶段没把多份文档拼合好、或拒答门槛设置过严，导致明明手里有信息却答不出。这两类是后续优化的重点。

**结论三：中小公司失败率小样本下偏高，值得抽样核查。** 周大福人寿 48.4%（n=31）、富卫人寿 47.8%（n=67）、中国人寿 46.5%（n=43）显著高于头部公司（友邦 24.1%、保诚 22.7%、万通 18.0%）。由于这些公司的样本量较小，这一现象只能表述为"小样本下偏高，值得抽样核查",不作为稳定结论。

## 6. 多轮会话观察

159 条多轮会话共 560 条追问消息：

| 追问类型 | 数量 | 占比 |
|---|---:|---:|
| 切到新主题（公司/类型变化） | 299 | 53.4% |
| 沿主题追细节 | 175 | 31.2% |
| 换问法重问 | 86 | 15.4% |

超过一半的追问其实是**切换新主题**——顾问把会话当作连续的工作台用，跨产品、跨公司换话题，并非真正意义上的多轮深入。

值得留意的细节：「**上一轮失败 → 顾问立刻换问法重问**」共出现 16 次，是检索失败影响顾问体感的可观察证据。比例不大（占多轮总量约 1%），说明绝大多数顾问不会在 AI 答不上来时反复纠缠，更可能直接放弃或换主题。

## 7. 优化方向（基于以上发现）

1. **入口提示顾问补充客户场景信息**。8 成以上首问不带客户参数（年龄/保额/年期/病史），AI 难以精准匹配。可在提问框做轻量提示——「补充客户年龄/预算/币种可获更精准答案」——或对抽象问题在首轮主动反问关键参数。这一步成本低、覆盖面最广。

2. **多产品对比、反向询问类是优先优化对象**。这两类失败率最高（26.6% / 32.7%），且 7 成以上失败发生在"已经检索到文档但生成阶段答不出"。建议针对对比类设计结构化输出模板（产品 × 维度的对照表），针对反向询问类核查输入分类的边界与生成阶段的拒答阈值。

3. **优惠信息高频出现，建议结构化呈现**。优惠/活动咨询占 22.4%，是仅次于产品详情的第二大需求；但优惠信息时效强、过滤后失败比例也高（21.4%）。把每家公司的当期优惠组织成结构化数据卡（产品 - 优惠类型 - 力度 - 截止日期），优于自由文本检索。

4. **中小公司资料覆盖待抽样核查**。周大福、富卫、中国人寿等失败率显著高于头部公司，但样本量小（n=31~67）。建议先抽样人工查这些公司在知识库中的文档数量与版本，再决定是否专项补充——而不是先大规模补资料。

---

## 8. 局限与后续工作

- **意图细类基于规则匹配**，未匹配率 17.3%，规则未覆盖的口语化表达可能被错分；后续可用语义聚类对未匹配样本做二次校准。
- **部分公司、产品样本量较小**（n < 50），分公司细看时已加注小样本警示，敏感结论需抽样人工复核。
- **失败诊断依赖 `nodes_history` 暴露的字段**；"有文档仍失败"（55%）的更细原因（生成模型遗漏文档信息、答案格式问题、拒答门槛设置过严）需要进一步抽样人工审阅生成节点的 prompt 与输出。
- **本次未涉及客户侧体验数据**（顾问是否将 AI 回答转发给客户、客户成单率、客户反馈），仅基于顾问 - AI 直接交互；若能拿到客户侧数据，可对"AI 回答的实际业务价值"做更完整的评估。

## 附录 A：检索失败的流水线节点级明细

正文 §5 以业务结论为主，本附录给出失败诊断的具体口径与拆分数据，便于复核。

**口径**：对每个失败会话遍历 assistant `metadata.nodes_history`，按是否进入检索分支、`*_search_docs_node.topk_docs` 是否非空、`*_docs_relevance_check_node` 中 `is_relevant=true` 的数量，归入下表 4 类之一。兼容 `simpleQA_*` 与 `discountQA_*` 两套节点前缀。

**全量 305 个失败会话的卡点分布**：

| 失败类别 | 数量 | 占失败 | 含义 |
|---|---:|---:|---|
| **有文档但回答仍失败** | 167 | **54.8%** | 召回正常、相关性也通过，但生成/评估环节失败 |
| 召回到但全被过滤 | 61 | 20.0% | 相关性判定阈值过严或 query 改写不充分 |
| 未走检索分支 | 53 | 17.4% | 被分类为闲聊/反向询问，跳过了 RAG |
| 召回为空 | 24 | 7.9% | 知识库中无可用文档 |

**高失败率问题类型 × 失败类别交叉**（多标签字段按 explode 计数，每行行内合计 = 该类型失败会话总数）：

| 问题类型 | 失败数 | 召回为空 | 全过滤掉 | **有文档仍失败** | 无检索步 |
|---|---:|---:|---:|---:|---:|
| 反向询问类 | 17 | 1 | 0 | **12** | 4 |
| 多业务对比 | 59 | 1 | 5 | **41** | 12 |
| 单业务细节咨询 | 224 | 19 | 44 | **129** | 32 |
| 优惠信息咨询 | 56 | 3 | 12 | 29 | 12 |

可以看到，无论问题类型粗细，"有文档仍失败"都是各类型下最大的失败子类——这是正文 §5 结论一的数据支撑。
